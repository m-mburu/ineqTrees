# LLM Instructions: Emit Training Metrics From CI Tuning

## Goal

Update the fusen source so `tune_ci_tree()` and `tune_ci_forest()` both emit consistent training and validation diagnostics in their tuning objects. After this change, `ci_collect_metrics(x, format = "tidy")` should return training rows for tree tuning as well as forest tuning.

Do not edit generated `R/*.R`, `man/*.Rd`, or `tests/testthat/*.R` files first. This is a fusen package. Edit the flat file:

```text
dev/model_selection.Rmd
```

Then regenerate with:

```r
fusen::inflate(
  flat_file = "dev/model_selection.Rmd",
  vignette_name = "Model selection",
  open_vignette = FALSE,
  check = FALSE,
  document = TRUE,
  overwrite = "yes",
  clean = "no"
)
```

## Current State

Relevant chunks in `dev/model_selection.Rmd`:

- `function-tune_ci_tree`: tree tuning implementation.
- `function-tune_ci_forest`: forest tuning implementation.
- `function-ci_tuning_collectors`: `ci_collect_metrics()` and tidy collector.
- `tests-tune_ci_tree` / `tests-tune_ci_forest`: package tests generated by fusen.

`tune_ci_forest()` already adds fold-level diagnostic columns to `fold_results`:

```r
train_gain
fold_validation_gain
train_relative_gain
fold_validation_relative_gain
relative_generalization_gap
train_root_impurity
validation_root_impurity
```

`tune_ci_tree()` currently only stores metric rows such as `validation_gain` and `relative_validation_gain`; it does not store training diagnostics. Because of that, `ci_collect_metrics(tree_tuning, format = "tidy")` cannot emit `train_gain` / `train_relative_gain` rows yet.

## Implementation Plan

### 1. Add tree training diagnostics inside `function-tune_ci_tree`

In the tree tuning task, after a fold fit succeeds and before metric rows are assembled, compute diagnostics for the fitted tree on both train and validation fold data.

Use existing helpers:

```r
.ci_data_root_impurity()
.ci_lookup_root_impurity()
.ci_score_validation_gain()
.ci_score_relative_validation_gain()
```

For each fitted tree / grid / fold / type, compute:

```r
train_root_impurity <- .ci_data_root_impurity(
  data = data[train_idx, , drop = FALSE],
  weights = weights[train_idx],
  rank_name = rank_name,
  outcome_name = outcome_name,
  type = this_type
)

validation_root_impurity <- .ci_lookup_root_impurity(root_table, fold, this_type)

train_gain <- .ci_score_validation_gain(
  fit = fit_result$value,
  new_data = data[train_idx, , drop = FALSE],
  rank_name = rank_name,
  outcome_name = outcome_name,
  weights = weights[train_idx],
  type = this_type,
  root_impurity = train_root_impurity
)

train_relative_gain <- .ci_score_relative_validation_gain(
  fit = fit_result$value,
  new_data = data[train_idx, , drop = FALSE],
  rank_name = rank_name,
  outcome_name = outcome_name,
  weights = weights[train_idx],
  type = this_type,
  root_impurity = train_root_impurity
)
```

Also compute validation diagnostics once per fold fit, so the diagnostic columns are available even when `metric` does not include both validation metrics:

```r
fold_validation_gain
fold_validation_relative_gain
```

These should use the validation fold and `validation_root_impurity`.

Wrap diagnostic scoring in `.ci_capture()` like the existing scoring code so warnings/errors are stored in `notes` rather than crashing tuning unnecessarily.

### 2. Add diagnostic columns to every tree `score_rows[[i]]`

When constructing each fold metric row for `tune_ci_tree()`, include the new diagnostic columns:

```r
train_gain = train_gain
fold_validation_gain = fold_validation_gain
train_relative_gain = train_relative_gain
fold_validation_relative_gain = fold_validation_relative_gain
relative_generalization_gap = train_relative_gain - fold_validation_relative_gain
train_root_impurity = train_root_impurity
validation_root_impurity = validation_root_impurity
```

Keep `score` as the metric-specific validation or prediction score. The diagnostic columns are repeated across metric rows for the same `grid_id` / `fold_id` / `type`, matching current `tune_ci_forest()` behavior.

### 3. Keep forest behavior aligned

Check `function-tune_ci_forest` after the tree change. Forest already emits the training diagnostic columns. If tree and forest column names differ, standardize on the existing forest names:

```r
train_gain
fold_validation_gain
train_relative_gain
fold_validation_relative_gain
relative_generalization_gap
train_root_impurity
validation_root_impurity
```

Do not rename these unless tests and `ci_collect_metrics(format = "tidy")` are updated at the same time.

### 4. Update `ci_collect_metrics(format = "tidy")`

The tidy collector already maps:

```r
train_gain -> .metric == "train_gain", .dataset == "training"
train_relative_gain -> .metric == "train_relative_gain", .dataset == "training"
```

After tree tuning emits those columns, no major collector redesign should be needed.

Optional improvement: add validation diagnostic fallback rows from `fold_validation_gain` / `fold_validation_relative_gain` when they exist, but avoid duplicating rows already represented by summary metrics `validation_gain` and `relative_validation_gain`.

### 5. Update tests in the flat file

In the tree collector test, add expectations that `tune_ci_tree()` fold results contain training diagnostics:

```r
diagnostic_cols <- c(
  "train_gain",
  "fold_validation_gain",
  "train_relative_gain",
  "fold_validation_relative_gain",
  "relative_generalization_gap",
  "train_root_impurity",
  "validation_root_impurity"
)
expect_true(all(diagnostic_cols %in% names(tuned$fold_results)))
```

Also assert the tidy collector returns training rows for tree tuning:

```r
tidy_metrics <- ci_collect_metrics(tuned, format = "tidy")
expect_true(all(c("validation_gain", "brier", "train_gain") %in% tidy_metrics$.metric))
expect_true(all(c("validation", "training") %in% tidy_metrics$.dataset))
```

Keep existing forest tests and make sure they still pass.

## Important Constraints

- This package uses fusen. The flat file is the source of truth.
- Prefer idiomatic `data.table`; the package imports `data.table`, so `.()`, `.SD`, joins, and `:=` are acceptable.
- Preserve existing default behavior of `ci_collect_metrics()`:

```r
ci_collect_metrics(x)
ci_collect_metrics(x, summarize = FALSE)
```

- `format = "tidy"` should remain opt-in.
- Do not break existing metric selection semantics: model selection still uses validation metrics, not training metrics.
